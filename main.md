# Safari AI Sidebar 项目说明

## 项目名称

Safari AI Sidebar for macOS

## 项目目标

构建一个仅面向 macOS Safari 的本地 AI 插件产品，以 `Sidebar` 作为主入口，在当前页面上下文中提供稳定的阅读理解与安全写入能力。

本项目当前目标不是上架 App Store，也不是做公开分发的大规模产品，而是：

- 先做出一个你自己能稳定使用的本地产品
- 支持少量用户进行小范围测试
- 将 OAuth 逆向登录能力视为既有黑盒，不在本轮重新开发

## 当前范围

首版只做以下两类能力：

1. 阅读与理解
- 总结当前页面
- 解释选中文本
- 针对当前页面进行问答
- 提取页面中的结构化信息

2. 安全写入
- 为当前聚焦输入框生成草稿
- 在用户确认后写入页面
- 绝不自动提交、绝不自动点击发送

## 当前实现说明

当前代码已经切到独立 Codex OAuth 架构，不再依赖本地反向代理。

- 浏览器按钮侧只保留最小账号模块：`登录`、`退出`、`当前账号`、`选择模型`
- 宿主 App 负责 Codex OAuth、单账户保存、模型拉取与默认模型管理
- OAuth 回调使用本地 `localhost` 监听，失败时保留 `127.0.0.1` 兜底

## 首版支持站点

- GitHub
- Gmail
- X
- Yahoo Mail

支持顺序按稳定性优先：

1. GitHub
2. Gmail
3. X
4. Yahoo Mail

## 技术路线

首版统一采用：

- 宿主 App：Swift + SwiftUI
- Safari 插件：Safari Web Extension + TypeScript
- Provider 层：本地 HTTP 接口
- OAuth 逆向登录：独立黑盒，通过本地 ProviderAdapter 调用

这样设计的原因：

- Safari 扩展与 macOS App 的边界清晰
- 宿主 App 与 Provider 解耦，后续可切换为官方 API
- TypeScript 适合 DOM 处理与页面注入
- Swift 适合做 macOS 本地 UI、权限控制、日志与分发

## 产品边界

本项目首版明确不做：

- 自动提交表单
- 自动点击发送按钮
- 自动登录网站
- 多标签页自主工作流
- 高风险网页操作
- 与 OAuth 黑盒深度耦合的前端逻辑

## 产品原则

### 1. 稳定优先

优先保证在核心站点上“稳定可用”，而不是追求大量功能。

### 2. 可回退

当页面识别或写入不稳定时，允许降级为只读模式或复制到剪贴板。

### 3. 安全写入

任何写入行为都必须满足：

- 可定位目标输入框
- 可在页面中高亮目标
- 可在 Sidebar 中明确展示目标
- 用户手动确认后才执行

### 4. Provider 可替换

逆向 OAuth 层不进入产品主架构核心。所有模型请求均通过统一 Provider 接口发送，以避免未来迁移成本失控。

## MVP 成功标准

首版成功不看功能多少，只看下面四件事是否成立：

1. Sidebar 可以稳定读取当前页面上下文
2. 页面总结与问答结果稳定返回
3. 当前输入框可被正确识别并安全写入
4. 在 GitHub、Gmail、X 上持续使用时没有明显崩溃或误操作

## 首版交付物

首版完成后应包含：

- 一个可安装的 macOS 宿主 App
- 一个可启用的 Safari Sidebar 插件
- 本地 Provider 适配层
- 基础会话保存能力
- 基础日志与调试导出
- 四个目标站点的最低可用适配

## 为什么值得做

这个项目的价值不在“再做一个聊天窗口”，而在：

- 将 AI 能力直接嵌入真实浏览器工作流
- 让页面理解与页面操作发生在统一上下文中
- 保留本地控制权与可调试性
- 形成一个可扩展的 Safari AI 产品骨架

## 后续演进方向

首版跑通后，可以逐步扩展：

- 多 Provider 支持
- 更细粒度的站点适配层
- 会话历史与页面知识缓存
- Prompt 模板与快捷动作
- 更强的结构化提取
- 更严格的权限与操作审计
